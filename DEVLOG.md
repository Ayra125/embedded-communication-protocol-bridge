## May 5, 2026 — MPU6050 I2C IMU Driver

**Commits:** `feat: add MPU6050 address define` · `feat: add mpu6050 header file` · `feat: implement MPU6050 driver` (×2) · `Add files via upload`

**Files:** `firmware/Peripherals/i2c/mpu6050.h` · `firmware/Peripherals/i2c/mpu6050.c`

The first real firmware day. Built the complete MPU6050 driver from scratch by reading the datasheet register map.

### Header (`mpu6050.h`)

Started by defining the I2C address and all register addresses I would need:

```c
#define MPU6050_ADDR        0x68
#define MPU6050_PWR_MGMT_1  0x6B
#define MPU6050_ACCEL_XOUT_H 0x3B
#define MPU6050_GYRO_XOUT_H  0x43
```

A key early lesson: the STM32 HAL expects a **left-shifted 8-bit address** (`MPU6050_ADDR << 1`) because it appends the R/W bit itself. The 7-bit device address from the datasheet is `0x68`, so the HAL call uses `0xD0`.

Also added include guards:
```c
#ifndef MPU6050_H
#define MPU6050_H
// ...
#endif
```
This prevents multiple-inclusion errors when headers are shared across files — a C discipline issue that bit me early and taught me why header guards exist.

### Implementation (`mpu6050.c`)

**Init** — The MPU6050 powers up in sleep mode by default. To wake it, write `0x00` to register `PWR_MGMT_1`:
```c
HAL_StatusTypeDef MPU6050_Init(I2C_HandleTypeDef *hi2c) {
    uint8_t data[2] = {MPU6050_PWR_MGMT_1, 0x00};
    return HAL_I2C_Master_Transmit(hi2c, MPU6050_ADDR << 1, data, 2, 500);
}
```

**ReadAccel / ReadGYRO** — Both use `HAL_I2C_Mem_Read` to burst-read 6 consecutive registers. The sensor stores each axis as a high byte followed by a low byte, so combining them requires bit shifting and OR:

```c
*x = (int16_t)(buffer[0] << 8 | buffer[1]);
*y = (int16_t)(buffer[2] << 8 | buffer[3]);
*z = (int16_t)(buffer[4] << 8 | buffer[5]);
```

**Key concepts learned today:**
- I2C register addressing and burst reads with `HAL_I2C_Mem_Read`
- Left-shifting device address by 1 for HAL compatibility
- Combining two 8-bit registers into a signed 16-bit value
- Datasheet register map navigation

## May 6, 2026 — SSD1306 OLED: Header & Constants

**Commit:** `feat: add SSD1306 header file`

**Files:** `firmware/Peripherals/i2c/sdd1306.h` · `firmware/Peripherals/i2c/ssd1306.c` (stub)

Moved on to the OLED display. Before touching the `.c` file, I spent this session reading the SSD1306 datasheet and building out the full header with command constants — a discipline I carried over from the MPU6050 work.

The SSD1306 has a large command set. Key ones I defined:

```c
#define SSD1306_ADDR            0x3C    // I2C address
#define SSD1306_DISP_ON         0xAF
#define SSD1306_SET_MUX_RATIO   0xA8
#define SSD1306_SET_CHARGE_PUMP 0x8D
#define SSD1306_CHARGE_PUMP_ON  0x14
#define SSD1306_COL_ADDR        0x21
#define SSD1306_PAGE_ADDR       0x22
#define SSD1306_CMD             0x00    // control byte: command follows
#define SSD1306_DATA            0x40    // control byte: data follows
```

The control byte protocol is unique to the SSD1306: every I2C transmission must begin with `0x00` (indicating a command) or `0x40` (indicating pixel data). This is different from the MPU6050 which uses the register address directly. Also created the `ssd1306.c` stub and committed it as a starting point for the next session.

## May 7, 2026 — SSD1306 OLED: Driver Skeleton & Clear Function

**Commits:** `feat: add SSD1306 driver skeleton` · `feat: implement SSD1306 Clear function`

**Files:** `firmware/Peripherals/i2c/ssd1306.c` · `firmware/Peripherals/i2c/sdd1306.h`

### Driver Skeleton

Implemented the core low-level transmission functions that everything else would build on:

```c
HAL_StatusTypeDef SSD1306_SendCommand(I2C_HandleTypeDef *hi2c, uint8_t cmd) {
    uint8_t data[2] = {SSD1306_CMD, cmd};
    return HAL_I2C_Master_Transmit(hi2c, SSD1306_ADDR << 1, data, 2, 500);
}

HAL_StatusTypeDef SSD1306_SendData(I2C_HandleTypeDef *hi2c, uint8_t *data, uint16_t size) {
    uint8_t buff[129];
    buff[0] = SSD1306_DATA;
    memcpy(&buff[1], data, size);
    return HAL_I2C_Master_Transmit(hi2c, SSD1306_ADDR << 1, buff, size + 1, 500);
}
```

Also added function prototypes to the header and stubbed out `SSD1306_Clear` and `SSD1306_WriteString`.

### Clear Function

The SSD1306 display is organized as 8 pages × 128 columns = 1024 bytes of RAM. To clear it, I need to set the column and page address windows, then send 1024 zero bytes:

```c
HAL_StatusTypeDef SSD1306_Clear(I2C_HandleTypeDef *hi2c) {
    uint8_t zeros[1024];
    SSD1306_SendCommand(hi2c, SSD1306_COL_ADDR);
    SSD1306_SendCommand(hi2c, 0);    // start column
    SSD1306_SendCommand(hi2c, 127);  // end column
    SSD1306_SendCommand(hi2c, SSD1306_PAGE_ADDR);
    SSD1306_SendCommand(hi2c, 0);    // start page
    SSD1306_SendCommand(hi2c, 7);    // end page
    memset(zeros, 0, sizeof(zeros));
    return SSD1306_SendData(hi2c, zeros, 1024);
}
```

Also refactored the `Init` sequence at this point — early stubs had incorrect function call signatures. Fixed to use `SSD1306_SendCommand(hi2c, ...)` with proper named constants instead of raw hex with wrong argument types.

**Key concepts learned today:**
- SSD1306 display RAM structure (8 pages × 128 columns)
- `memset` for zeroing buffers
- `memcpy` for prepending a control byte to a data payload
- Importance of naming constants — init code went from unreadable hex to self-documenting commands

## May 8, 2026 — SSD1306 Complete + W25Q128 SPI Flash Driver

**Commits:** `feat: complete SSD1306 driver` · `feat: complete W25Q128 SPI driver`

**Files:** `firmware/Peripherals/i2c/ssd1306.c` · `firmware/Peripherals/spi/w25q128.c` · `firmware/Peripherals/spi/w25q128.h`

A major day — completed two drivers.

### SSD1306: Font Table & WriteString

The final piece of the display driver was text rendering. Embedded systems don't have fonts or OS graphics APIs — you have to define every character as a pixel bitmap yourself.

I embedded a 5×7 font table as a static 2D byte array covering ASCII characters 32–90:

```c
static const uint8_t font[][5] = {
    {0x00, 0x00, 0x00, 0x00, 0x00}, // 32 space
    {0x00, 0x00, 0x5F, 0x00, 0x00}, // 33 !
    {0x7E, 0x11, 0x11, 0x11, 0x7E}, // 65 A
    // ...
};
```

Each byte in a column represents 8 vertical pixels (1 bit per row). `WriteString` loops through each character, indexes into the font table, and sends the 5 pixel bytes to the display via `SendData`.

Also fixed a bug from the May 7 skeleton: the Init function was calling `SSD1306(*hi2c, ...)` (wrong — treated the function as a generic macro) instead of `SSD1306_SendCommand(hi2c, ...)`. Corrected and expanded the full initialization sequence from the datasheet application note.

### W25Q128 SPI Flash Driver

Switched protocols for the first time — from I2C to SPI. The W25Q128 is a 16MB flash memory chip. Key differences from I2C:

- **No addressing** — instead of an I2C address, the chip is selected by pulling a CS (Chip Select) pin LOW manually before each transaction
- **Erase before write** — flash memory cells can only be programmed from 1→0. To write new data, you must first erase the sector (sets all bits back to 1), then write.
- **Write Enable Latch (WEL)** — the chip ignores write commands unless you first send a Write Enable command. After every erase or write, the latch resets automatically.

Implemented five functions:

```c
// Wake from power-down mode
W25Q128_Init(SPI_HandleTypeDef *hspi)

// Poll Status Register 1 until BUSY bit clears
W25Q128_WaitBusy(SPI_HandleTypeDef *hspi)

// Erase a 4KB sector at the given address
W25Q128_EraseSector(SPI_HandleTypeDef *hspi, uint32_t address)

// Write data via Page Program command
W25Q128_Write(SPI_HandleTypeDef *hspi, uint32_t address, uint8_t *data, uint16_t size)

// Read data from address
W25Q128_Read(SPI_HandleTypeDef *hspi, uint32_t address, uint8_t *data, uint16_t size)
```

24-bit address splitting across 3 bytes was a key pattern used throughout:
```c
(address >> 16) & 0xFF,   // upper byte
(address >> 8)  & 0xFF,   // middle byte
address         & 0xFF,   // lower byte
```

**Key concepts learned today:**
- SPI CS pin manual management (GPIO pull LOW → transmit → pull HIGH)
- Flash memory erase-before-write constraint
- Write Enable Latch protocol
- Bitwise AND masking (`& 0x01`) to read individual status register bits
- 24-bit address splitting with bit shifts
